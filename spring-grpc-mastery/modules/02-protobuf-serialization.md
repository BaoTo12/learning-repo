# Module 02: Protocol Buffers & Schema Evolution

## 1. What Problem This Module Solves
Microservice ecosystems require a way to share data models across services. Text-based formats (like JSON) introduce severe production concerns:
*   **High CPU Overhead**: Text scanning and mapping to Java objects via reflection consumes CPU cycles and triggers garbage collection.
*   **API Drift**: Services can easily deploy breaking changes (like renaming a field) without triggering compilation errors in client applications.
*   **Payload Inflation**: Repeated metadata keys (e.g. `{"order_id": 101}`) inflate network payloads, slowing down high-frequency inter-service transfers.

Protocol Buffers (Protobuf) solves this by separating message definitions from serialization code, enforcing a strict binary format that compiles directly into native Java classes.

---

## 2. Why Chosen Over Alternatives
Protobuf uses a **binary tag-value encoding scheme**. Instead of serializing field names, fields are assigned integer **tags**. The serialized payload contains only these tag identifiers and their raw values.

```
JSON payload (38 bytes):
{"userId":101,"email":"user@test.com"}

Protobuf binary payload (18 bytes):
08 65 12 0d 75 73 65 72 40 74 65 73 74 2e 63 6f 6d
```
On the wire, tag `1` (userId) encodes as `08` (Tag 1, Wire Type 0), and tag `2` (email) encodes as `12` (Tag 2, Wire Type 2). This tag layout minimizes serialization overhead.

## 3. Protobuf Syntax & Typing (Proto3)

Here is a comprehensive schema incorporating enums, repeated fields, nested messages, maps, `oneof` unions, and Google Well-Known Types (WKTs):

```protobuf
syntax = "proto3";

package api.user.v1;

import "google/protobuf/timestamp.proto";
import "google/protobuf/any.proto";

option java_multiple_files = true;
option java_package = "com.example.grpc.user.v1";

enum UserStatus {
  USER_STATUS_UNSPECIFIED = 0;
  USER_STATUS_ACTIVE = 1;
  USER_STATUS_SUSPENDED = 2;
}

message UserProfile {
  int32 user_id = 1;             // Tag 1 (Int32 Varint)
  string email = 2;              // Tag 2 (Length-delimited string)
  repeated string roles = 3;     // Tag 3 (Array list)
  UserStatus status = 4;         // Tag 4 (Enum value)

  message Address {
    string street = 1;
    string city = 2;
  }
  repeated Address addresses = 5; // Tag 5 (Nested message array)

  map<string, string> attributes = 6; // Tag 6 (Map payload)

  oneof auth_method {
    string oauth_provider = 7;
    string password_hash = 8;
  }

  google.protobuf.Timestamp last_login = 9; // Well-Known Type
  google.protobuf.Any custom_payload = 10;   // Well-Known Type for arbitrary extensions
}
```

## 4. Schema Evolution: Backward & Forward Compatibility

Schema evolution is the process of updating message definitions without breaking existing deployments.

*   **Backward Compatibility**: New code can read payloads generated by old code. (e.g. New client parses old server message).
*   **Forward Compatibility**: Old code can read payloads generated by new code. (e.g. Old client parses new server message).

### Crucial Evolution Rules
1.  **NEVER change the tag number of an existing field**. Tags are the primary identifiers on the wire.
2.  **If you delete a field, mark its tag and name as `reserved`**. This prevents future developers from reusing them and corrupting legacy payloads.
    ```protobuf
    message UserProfile {
      reserved 2, 7 to 8;
      reserved "email", "oauth_provider", "password_hash";
    }
    ```
3.  **Use `oneof` with caution**. Moving existing fields inside or out of a `oneof` block is a breaking change at the wire layer.

## 5. Spring Boot Utility Class for Protobuf Manipulation

This class demonstrates how to serialize/deserialize protobuf objects and manipulate Well-Known Types in a Spring Boot application:

```java
package com.example.grpc.user;

import com.example.grpc.user.v1.UserProfile;
import com.example.grpc.user.v1.UserStatus;
import com.google.protobuf.Any;
import com.google.protobuf.Timestamp;
import com.google.protobuf.util.JsonFormat;
import java.io.IOException;
import java.time.Instant;

public class ProtobufSerializationHelper {

    /**
     * Converts a Java Instant into a Protobuf Timestamp.
     */
    public static Timestamp toTimestamp(Instant instant) {
        return Timestamp.newBuilder()
            .setSeconds(instant.getEpochSecond())
            .setNanos(instant.getNano())
            .build();
    }

    /**
     * Converts a Protobuf message directly into JSON string representation.
     */
    public static String toJson(UserProfile profile) throws IOException {
        return JsonFormat.printer()
            .includingDefaultValueFields()
            .print(profile);
    }

    /**
     * Pack a custom message inside a google.protobuf.Any container.
     */
    public static Any packPayload(com.google.protobuf.Message message) {
        return Any.pack(message);
    }

    public static void main(String[] args) throws Exception {
        UserProfile user = UserProfile.newBuilder()
            .setUserId(99)
            .setEmail("test@example.com")
            .setStatus(UserStatus.USER_STATUS_ACTIVE)
            .setLastLogin(toTimestamp(Instant.now()))
            .build();

        // 1. Serialize to binary byte array
        byte[] binaryData = user.toByteArray();
        System.out.println("Serialized Binary Bytes count: " + binaryData.length);

        // 2. Deserialize from binary back to object
        UserProfile parsedUser = UserProfile.parseFrom(binaryData);
        System.out.println("Parsed User ID: " + parsedUser.getUserId());

        // 3. Print JSON conversion
        System.out.println("JSON representation:\n" + toJson(parsedUser));
    }
}
```

## 6. Common Mistakes and Anti-Patterns
*   **Reusing Deprecated Tags**: Reallocating tag `2` to a different type (e.g. changing from `string email` to `int32 loyaltyPoints`). Old services attempting to parse this will crash or produce garbage data.
*   **Assuming Required Validation**: Proto3 removes the `required` keyword from proto2. All fields are optional. Your Java code must explicitly validate fields (e.g., check `hasLastLogin()` before accessing a timestamp) to prevent `NullPointerException` errors.

## 7. Interview Questions

### Q1: Why did the creators of Proto3 remove the `required` keyword that was present in Proto2?
**Answer**: 
The `required` keyword was removed because it is highly disruptive for API evolution. If you define a field as `required`, and later decide to stop using or deprecate it, you cannot remove it from the proto schema without breaking all legacy clients that expect that field to be present. 
Removing `required` forces developers to design backward-compatible systems, handling missing fields at the application logic layer using default values instead.

### Q2: What is the purpose of `google.protobuf.Any`, and how does it work under the hood?
**Answer**: 
*   `google.protobuf.Any` is a Well-Known Type used to embed arbitrary serialized messages without having their definitions compile-time bound to the parent message.
*   **Under the hood**, it contains two fields: a string URL (`type_url`) representing the type path of the message (e.g. `type.googleapis.com/api.user.v1.UserProfile`), and a raw byte array (`value`) containing the binary payload. The client uses the `type_url` to look up the correct protobuf class and deserialize the raw bytes dynamically using `Any.unpack()`.
