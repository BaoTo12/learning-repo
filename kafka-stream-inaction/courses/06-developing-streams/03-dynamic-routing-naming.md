# Module 03: Topology Naming, Branching & Dynamic Routing

As stream processing applications evolve, topologies grow in complexity. You must frequently route events down different processing paths, merge streams back together, inspect processing graphs, and dynamically choose output destination topics. 

This module covers stream branching and merging, topology inspection, the critical importance of naming topology nodes, and implementing dynamic record routing using headers and metadata.

---

## 1. Branching and Merging Streams

### 1.1 Splitting Streams into Branches
Rather than filtering the same source stream multiple times (which creates separate consumer allocations), you can use the `split()` operator to split a stream into multiple branches based on predicates.

The branching model utilizes the `BranchedKStream` API:
*   **`branch(Predicate, Branched)`**: Evaluates a predicate. If `true`, routes the record to the specified branch.
*   **`defaultBranch(Branched)`**: A fallback branch for records that did not match any of the previous predicates.
*   **`noDefaultBranch()`**: Drops any records that do not match the branch predicates.

#### Branch Types (Consumer vs. Function vs. Alias)
Depending on how you want to handle the branched stream:
1.  **`Branched.withConsumer`**: Executes in-place logic (e.g. writing to a sink topic directly). This is a terminal action; the branch is not registered in the returned Map.
2.  **`Branched.withFunction`**: Evaluates a transformation function on the branch KStream and returns the result, which is added to the returned Map.
3.  **`Branched.as(String)`**: Serves as an alias. The raw branch stream is added to the returned Map using the concatenated name.

```java
Predicate<String, RetailPurchase> isCoffee = (k, v) -> v.department().equalsIgnoreCase("coffee");
Predicate<String, RetailPurchase> isElectronics = (k, v) -> v.department().equalsIgnoreCase("electronics");

Map<String, KStream<String, RetailPurchase>> branchedStreams = purchaseStream.split(Named.as("split-"))
    .branch(isCoffee, Branched.as("coffee"))
    .branch(isElectronics, Branched.withFunction(stream -> stream.mapValues(this::applyDiscount), "discount-electronics"))
    .defaultBranch(Branched.as("retail"));

// Access the streams from the returned map
KStream<String, RetailPurchase> coffeeStream = branchedStreams.get("split-coffee");
KStream<String, RetailPurchase> retailStream = branchedStreams.get("split-retail");
```

### 1.2 Merging Streams
To combine different streams with the same key/value types, use the `KStream.merge` operator.

```java
KStream<String, RetailPurchase> mergedStream = coffeeStream.merge(retailStream);
```
> [!NOTE]
> `merge()` does not guarantee strict inter-stream ordering (records from one stream may interleave with another), but it does preserve ordering within each individual stream.

## 2. Naming Topology Nodes

By default, Kafka Streams automatically generates names for topology nodes:
`KSTREAM-SOURCE-0000000000` -> `KSTREAM-MAPVALUES-0000000001` -> `KSTREAM-SINK-0000000002`

```java
TopologyDescription description = builder.build().describe();
System.out.println(description.toString());
```

### 2.1 The Operational Risk of Autogenerated Names
Autogenerated names are numbered sequentially based on their declaration order. If you add an operator (e.g., a simple `.peek()` for debugging) at the beginning of your code:
*   All downstream names shift (e.g., `KSTREAM-SINK-0000000002` becomes `0000000003`).
*   **State Store Corruptions**: Internal state stores and changelog topics are bound to these operator names. Shifting the numbers will cause the application to try to read the wrong changelogs, leading to state corruption, data loss, or partition rebalance crashes.

### 2.2 Production Best Practice: Explicit Naming
Always name your nodes using `Named.as()`, `Consumed.as()`, and `Produced.as()`/`withName()`.

```java
builder.stream("src-topic", Consumed.with(stringSerde, stringSerde).withName("Source-Inbound"))
       .mapValues(value -> value.toUpperCase(), Named.as("Processor-Uppercase"))
       .to("out-topic", Produced.with(stringSerde, stringSerde).withName("Sink-Outbound"));
```
This produces a stable, descriptive topology regardless of code changes:
```
Topologies:
   Sub-topology: 0
    Source: Source-Inbound (topics: [src-topic]) --> Processor-Uppercase
    Processor: Processor-Uppercase (stores: []) --> Sink-Outbound <-- Source-Inbound
    Sink: Sink-Outbound (topic: out-topic) <-- Processor-Uppercase
```

## 3. Dynamic Message Routing

Occasionally, records must be routed to different topics dynamically based on record contents, headers, or metadata rather than predefined static configurations.

### 3.1 Implementing a `TopicNameExtractor`
To route records dynamically, pass a `TopicNameExtractor` to the `to()` operator.

```java
package com.enterprise.streams.routing;

import org.apache.kafka.streams.processor.RecordContext;
import org.apache.kafka.streams.processor.TopicNameExtractor;

public class DepartmentTopicExtractor implements TopicNameExtractor<String, RetailPurchase> {

    @Override
    public String extract(String key, RetailPurchase value, RecordContext recordContext) {
        String department = value.department();
        if (department.equalsIgnoreCase("coffee") || department.equalsIgnoreCase("electronics")) {
            return "zmart-dept-" + department.toLowerCase();
        }
        return "zmart-default-purchases";
    }
}
```

### 3.2 Reading Context and Headers via `RecordContext`
The `RecordContext` parameter provides access to the record's broker metadata:
*   `recordContext.topic()`: Originating topic name.
*   `recordContext.partition()`: Originating partition number.
*   `recordContext.offset()`: Broker offset.
*   `recordContext.timestamp()`: Record timestamp.
*   `recordContext.headers()`: Custom metadata headers.

#### Routing via Header Example:
Below, we extract a routing destination dynamically from a custom metadata header:

```java
package com.enterprise.streams.routing;

import org.apache.kafka.common.header.Header;
import org.apache.kafka.common.header.Headers;
import org.apache.kafka.streams.processor.RecordContext;
import org.apache.kafka.streams.processor.TopicNameExtractor;

import java.nio.charset.StandardCharsets;

public class HeaderRoutingExtractor implements TopicNameExtractor<String, RetailPurchase> {

    private final String defaultTopic;

    public HeaderRoutingExtractor(String defaultTopic) {
        this.defaultTopic = defaultTopic;
    }

    @Override
    public String extract(String key, RetailPurchase value, RecordContext recordContext) {
        Headers headers = recordContext.headers();
        if (headers != null) {
            Header routingHeader = headers.lastHeader("X-Target-Region");
            if (routingHeader != null && routingHeader.value() != null) {
                String targetRegion = new String(routingHeader.value(), StandardCharsets.UTF_8);
                return "purchases-" + targetRegion.toLowerCase();
            }
        }
        return defaultTopic;
    }
}
```

#### Integrating the Extractor in Topology:
```java
purchaseStream.to(
    new HeaderRoutingExtractor("purchases-unknown"),
    Produced.with(stringSerde, purchaseSerde).withName("Sink-Dynamic-Routing")
);
```

> [!WARNING]
> Dynamic routing extracts names at runtime. **Kafka Streams does not automatically create these dynamic topics.** You must ensure that all potential destination topics exist on the broker cluster ahead of time. If a record returns a topic name that does not exist, the producer thread will block or crash.
